TÊN DỰ ÁN: DataPilot

PITCH:
"Hỏi dữ liệu bằng tiếng tự nhiên — DataPilot dịch câu hỏi của bạn thành SQL,
chạy truy vấn trên SQLite, và trả lời bằng lời văn rõ ràng, kèm báo cáo tuần

TẠI SAO ĐẠT 10/10 (giải thích kỹ thuật):

Thang điểm theo README:
  - Old school, ko smart            → max 7
  - Smart, ko LLM                   → max 8.5
  - Smart, LLM wrapper              → max 9
  - Smart, LLM là main program,
    cung cấp tools/API/MCP/skills   → max 10

DataPilot đạt 10/10 nhờ 3 lớp kỹ thuật thật sự:

1. TOOL-USE LOOP THẬT SỰ (không phải fake):
   Claude không chỉ sinh SQL và trả về. Claude *chạy tool* execute_sql, *nhận
   kết quả thật*, rồi *tiếp tục suy luận* dựa trên kết quả đó. Đây là agentic
   loop chuẩn — LLM là bộ não điều phối tool execution.

2. MCP SERVER CHUẨN GIAO THỨC:
   Backend expose dữ liệu qua Model Context Protocol server. Client AI đọc
   resources (schema bảng) và gọi tools (execute_sql) thông qua JSON-RPC —
   đúng cách Anthropic thiết kế cho AI agent truy cập data.

3. SKILL = WORKFLOW CÓ TRẠNG THÁI:
   Weekly report skill không chỉ gọi một tool — nó là chuỗi tool calls có
   logic: lấy danh sách bảng → phân tích dữ liệu tuần → sinh báo cáo →
   format kết quả. Đây là "compound tool use."

Kết hợp ba lớp này = 10/10. Thiếu MCP hoặc thiếu skill loop = 9/10.

LUỒNG TOOL-USE CỦA LLM (chi tiết):

  User: "Top 5 sản phẩm bán chạy nhất là gì?"

  Step 1: Orchestrator gửi message + tool definitions đến Claude

  Step 2: Claude quyết định cần biết schema trước
          -> tool_use { name: "list_tables" }

  Step 3: Orchestrator gọi MCP tool list_tables
          -> kết quả: ["Categories","Customers","OrderDetails",
                       "Orders","Products","Suppliers",...]

  Step 4: Claude quyết định cần schema OrderDetails và Products
          -> tool_use { name: "describe_table", input: {table: "OrderDetails"} }

  Step 5: Orchestrator gọi MCP, nhận column definitions

  Step 6: Claude đủ thông tin, sinh SQL
          -> tool_use { name: "execute_sql", input: { sql: "SELECT ..." } }

  Step 7: Safety layer kiểm tra -> chỉ có SELECT -> cho phép
          -> kết quả: [{ProductName: "Chai", total: 120}, ...]

  Step 8: Claude tổng hợp câu trả lời tự nhiên
          -> "Top 5 sản phẩm bán chạy: 1. Chai — 120 đơn vị..."

  Orchestrator -> FastAPI -> Frontend -> Hiển thị

================================================================================
  PHẦN 9: SECURITY — SQL READ-ONLY (3 LỚP BẢO VỆ)
================================================================================

LỚP 1: SQLite Connection Mode (Hardware-level)
  uri = f"file:{db_path}?mode=ro"
  conn = sqlite3.connect(uri, uri=True)
  -> SQLite raise OperationalError cho mọi write operation ở file system level

LỚP 2: SQL Parser (Logic-level) — dùng sqlglot
  - Keyword check: first_word in BLOCKED_KEYWORDS
    Blocked: insert, update, delete, drop, create, alter, truncate, replace
  - AST parse: sqlglot.parse(sql, dialect="sqlite")
    Chặn statement types: Insert, Update, Delete, Drop, Create, Alter, Truncate

LỚP 3: System Prompt Instruction (AI-level)
  QUAN TRỌNG trong system prompt:
  "Chỉ được dùng execute_sql với SELECT.
   TUYỆT ĐỐI KHÔNG sinh INSERT/UPDATE/DELETE/DROP/CREATE/ALTER.
   Nếu user yêu cầu xóa/sửa dữ liệu — từ chối lịch sự."
